32 ## Why is my debug build on Windows so slow?
33 
34 `EnTT` is an experimental project that I also use to keep me up-to-date with the
35 latest revision of the language and the standard library. For this reason, it's
36 likely that some classes you're working with are using standard containers under
37 the hood.<br/>
38 Unfortunately, it's known that the standard containers aren't particularly
39 performing in debugging (the reasons for this go beyond this document) and are
40 even less so on Windows apparently. Fortunately this can also be mitigated a
41 lot, achieving good results in many cases.
42 
43 First of all, there are two things to do in a Windows project:
44 
45 * Disable the [`/JMC`](https://docs.microsoft.com/cpp/build/reference/jmc)
46  option (_Just My Code_ debugging), available starting in Visual Studio 2017
47  version 15.8.
48 
49 * Set the [`_ITERATOR_DEBUG_LEVEL`](https://docs.microsoft.com/cpp/standard-library/iterator-debug-level)
50  macro to 0. This will disable checked iterators and iterator debugging.
51 
52 Moreover, the macro `ENTT_DISABLE_ASSERT` should be defined to disable internal
53 checks made by `EnTT` in debug. These are asserts introduced to help the users,
54 but require to access to the underlying containers and therefore risk ruining
55 the performance in some cases.
56 
57 With these changes, debug performance should increase enough for most cases. If
58 you want something more, you can can also switch to an optimization level `O0`
59 or preferably `O1`.

61 ## How can I represent hierarchies with my components?
62 
63 This is one of the first questions that anyone makes when starting to work with
64 the entity-component-system architectural pattern.<br/>
65 There are several approaches to the problem and whatâ€™s the best one depends
66 mainly on the real problem one is facing. In all cases, how to do it doesn't
67 strictly depend on the library in use, but the latter can certainly allow or
68 not different techniques depending on how the data are laid out.
69 
70 I tried to describe some of the techniques that fit well with the model of
71 `EnTT`. [Here](https://skypjack.github.io/2019-06-25-ecs-baf-part-4/) is the
72 first post of a series that tries to explore the problem. More will probably
73 come in future.
74 
75 Long story short, you can always define a tree where the nodes expose implicit
76 lists of children by means of the following type:
77 
78 ```cpp
79 struct relationship {
80  std::size_t children{};
81  entt::entity first{entt::null};
82  entt::entity prev{entt::null};
83  entt::entity next{entt::null};
84  entt::entity parent{entt::null};
85  // ... other data members ...
86 };
87 ```
88 
89 The sort functionalities of `EnTT`, the groups and all the other features of the
90 library can help then to get the best in terms of data locality and therefore
91 performance from this component.

93 ## Custom entity identifiers: yay or nay?
94 
95 Custom entity identifiers are definitely a good idea in two cases at least:
96 
97 * If `std::uint32_t` isn't large enough as an underlying type.
98 * If you want to avoid conflicts when using multiple registries.
99 
100 These identifiers are nothing more than enum classes with some salt.<br/>
101 To simplify the creation of new identifiers, `EnTT` provides the macro
102 `ENTT_OPAQUE_TYPE` that accepts two arguments:
103 
104 * The name you want to give to the new identifier (watch out for namespaces).
105 * The underlying type to use (either `std::uint16_t`, `std::uint32_t`
106  or `std::uint64_t`).
107 
108 In fact, this is the definition of `entt::entity`:
109 
110 ```cpp
111 ENTT_OPAQUE_TYPE(entity, std::uint32_t)
112 ```
113 
114 The use of this macro is highly recommended, so as not to run into problems if
115 the requirements for the identifiers should change in the future.

117 ## Warning C4307: integral constant overflow
118 
119 According to [this](https://github.com/skypjack/entt/issues/121) issue, using a
120 hashed string under VS could generate a warning.<br/>
121 First of all, I want to reassure you: it's expected and harmless. However, it
122 can be annoying.
123 
124 To suppress it and if you don't want to suppress all the other warnings as well,
125 here is a workaround in the form of a macro:
126 
127 ```cpp
128 #if defined(_MSC_VER)
129  #define HS(str)\
130  __pragma(warning(push))\
131  __pragma(warning(disable:4307))\
132  entt::hashed_string{str}\
133  __pragma(warning(pop))
134 #else
135  #define HS(str) entt::hashed_string{str}
136 #endif
137 ```
138 
139 With an example of use included:
140 
141 ```cpp
142 constexpr auto identifier = HS("my/resource/identifier");
143 ```
144 
145 Thanks to [huwpascoe](https://github.com/huwpascoe) for the courtesy.

147 ## Warning C4003: the min, the max and the macro
148 
149 On Windows, a header file defines two macros `min` and `max` which may result in
150 conflicts with their counterparts in the standard library and therefore in
151 errors during compilation.
152 
153 It's a pretty big problem but fortunately it's not a problem of `EnTT` and there
154 is a fairly simple solution to it.<br/>
155 It consists in defining the `NOMINMAX` macro before to include any other header
156 so as to get rid of the extra definitions:
157 
158 ```cpp
159 #define NOMINMAX
160 ```
161 
162 Please refer to [this](https://github.com/skypjack/entt/issues/96) issue for
163 more details.
